# Product

## Register

product

## Users

主要用户是飞书环境中的团队成员、项目负责人和需要快速沉淀汇报材料的协作者。他们的需求通常从 IM 群聊或单聊中出现，希望用自然语言触发任务，让系统自动完成意图捕捉、任务规划、文档生成、演示稿生成和交付回流。

用户使用这个界面时，不是在从零创作内容，而是在查看 Agent 已经理解了什么、当前执行到哪一步、哪里需要确认，以及最终交付物是否可用。页面需要适合飞书内嵌环境、桌面端和窄屏视口，降低学习成本，避免让普通业务用户看到过多技术字段。

## Product Purpose

Agent-Pilot 是一个基于 IM 的办公协同智能助手，目标是把“从聊天需求到文档和演示稿交付”的重复流程交给 AI Agent 驱动。Agent 负责理解意图、拆解步骤、调用大模型和飞书 API 执行业务动作；GUI 是仪表盘与辅助操作面，负责展示进度、提供必要确认和预览交付结果。

成功状态不是把页面做成复杂工作台，而是让用户能清楚判断任务是否正常推进、当前是否需要介入、产物是否已经交付，并能在飞书上下文里自然回到协作流程。

## Brand Personality

语气应当克制、可信、清晰。界面表达接近飞书生态里的业务工具：低噪音、效率优先、信息层级稳定，不制造强品牌感，也不把 AI 能力包装成夸张的营销叙事。

核心感受是“可靠的协作助手”：让用户感觉任务链路可见、确认动作明确、结果可追踪，同时保持足够轻量，不干扰飞书原本的工作流。

## Anti-references

不要做成通用 agent dashboard、运维控制台或调试台。默认避免右侧工作台、事件流、审计日志、trace 面板、技术元数据、任务 ID、请求 ID、创建时间和更新时间等普通用户不需要的信息。

不要把自然语言入口放在页面输入框里。当前产品语义是从飞书聊天 API 获取输入，页面负责展示、确认和预览，除非明确需要本地模拟或调试。

不要使用重装饰、强品牌渐变、大面积深色科技感、过度卡片化布局或营销页面式 hero。视觉上应与飞书蓝白灰、细分割线、轻阴影和紧凑商务风保持一致。

## Design Principles

1. Agent 主驾驶，界面只展示必要状态和确认动作。
2. 飞书上下文优先，所有信息都应服务于 IM 触发、任务推进和交付回流。
3. 隐藏技术细节，优先使用业务语言解释状态、步骤和产物。
4. 保持信息架构稳定，除非用户明确要求，不主动加入工作台、日志或调试模块。
5. 先满足赛题主链路，再考虑装饰性增强或可选能力。

## Accessibility & Inclusion

默认按 WCAG 2.1 AA 的可读性和键盘可用性目标处理。界面需要支持中文长文本、窄屏单列布局、清晰焦点状态、足够的文本对比度，并尊重 `prefers-reduced-motion`，避免依赖颜色作为唯一状态表达。
