# 工作台首屏快照联调说明

## 目标

解决从飞书卡片点击进入工作台时，页面先出现“原始无内容/空白态”，等待一小段时间后才渲染真实内容的问题。

前端进入工作台后，应先请求任务快照接口，并用快照数据完成首屏渲染；SSE 只负责后续实时更新，不应作为首屏内容的唯一来源。

## 推荐加载顺序

1. 从 URL 读取 `taskId`。
2. 立即请求快照接口：`GET /api/v1/tasks/{taskId}/workspace`。
3. 快照返回后，用响应体一次性渲染标题、进度、步骤、确认区、预览区、时间轴和产物区。
4. 快照渲染完成后，再建立 SSE：`GET /api/v1/tasks/{taskId}/workspace/stream`。
5. 收到 SSE 事件后，用最新 `TaskWorkspaceView` 替换当前页面状态。

## 接口

### 首屏快照

```http
GET /api/v1/tasks/{taskId}/workspace
```

用途：工作台首屏唯一必需数据源。

前端不应在这个接口返回前展示“无任务内容”的空状态。可以展示加载骨架屏，但不能展示“暂无内容/未生成计划/无预览”等最终态文案。

### 实时同步

```http
GET /api/v1/tasks/{taskId}/workspace/stream
```

事件类型：

- `snapshot`：订阅成功后服务端立即推送一次当前快照。
- `workspace`：任务状态变化后的实时快照。

SSE 数据结构与首屏快照接口一致，都是完整 `TaskWorkspaceView`，不是 patch。

## 关键字段

### 顶层状态

```json
{
  "taskId": "task-1",
  "title": "校招宣讲PPT",
  "status": "WAIT_CONFIRM",
  "displayStatus": "等待确认",
  "inputSummary": "帮我们生成一份校招宣讲PPT",
  "contextText": "【IM任务输入】..."
}
```

渲染建议：

- 页面标题优先使用 `title`。
- 副标题/需求摘要使用 `inputSummary`。
- 调试详情或展开区再使用 `contextText`，不要首屏大段展示。

### 稳定进度

```json
{
  "progress": {
    "percent": 60,
    "phaseIndex": 4,
    "totalPhases": 6,
    "phase": "confirm2",
    "label": "预览待确认"
  }
}
```

前端进度条必须优先使用 `progress.percent`，不要再自行根据 `steps[].status` 推导百分比。

当前阶段约定：

| phase | percent | 含义 |
|---|---:|---|
| `created` | 10 | 任务已创建 |
| `planned` | 25 | 计划已生成 |
| `confirm1` | 40 | 需求待确认 |
| `preview_generating` | 50 | 需求已确认，正在生成预览 |
| `confirm2` | 60 | 预览待确认 |
| `running` | 80 | 正在正式创建飞书产物或继续执行 |
| `delivered` | 100 | 已完成 |
| `failed` | 100 | 失败 |
| `cancelled` | 100 | 已取消 |

注意：同一个步骤在 confirm2 时仍可能是 `WAIT_CONFIRM`，这是业务状态，不代表进度回退。前端应以 `progress.phase` 和 `steps[].phase` 为准展示阶段。

### 步骤列表

```json
{
  "steps": [
    {
      "stepId": "D_SLIDES",
      "code": "D_SLIDES",
      "name": "创建宣讲 PPT",
      "action": "创建宣讲 PPT",
      "status": "WAIT_CONFIRM",
      "displayStatus": "预览待确认",
      "phase": "confirm2",
      "requiresConfirm": true,
      "active": true
    }
  ]
}
```

渲染建议：

- 展示状态文案用 `displayStatus`。
- 高亮当前步骤用 `active`。
- 区分确认阶段用 `phase`，不要只看 `status`。

### 确认区

```json
{
  "confirmation": {
    "waiting": true,
    "phase": "confirm2",
    "stepId": "D_SLIDES",
    "title": "预览已生成，请确认是否正式创建飞书产物",
    "description": "你可以查看预览并精修；确认后，Agent 将正式创建飞书产物。",
    "action": "创建宣讲 PPT",
    "artifactType": "PPT",
    "theme": "business",
    "previewReady": true,
    "preview": {
      "available": true,
      "type": "slides",
      "title": "宣讲 PPT",
      "theme": "business",
      "stepId": "D_SLIDES",
      "data": {}
    }
  }
}
```

按钮行为：

确认：

```http
POST /api/v1/tasks/{taskId}/workspace/confirm
Content-Type: application/json

{
  "stepId": "D_SLIDES",
  "approved": true,
  "source": "workspace",
  "clientId": "desktop-web"
}
```

取消：

```http
POST /api/v1/tasks/{taskId}/workspace/cancel
Content-Type: application/json

{
  "stepId": "D_SLIDES",
  "comment": "用户取消",
  "source": "workspace",
  "clientId": "desktop-web"
}
```

`clientId` 建议按端区分，例如：

- `desktop-web`
- `mobile-web`

### 预览区

```json
{
  "preview": {
    "available": true,
    "type": "slides",
    "title": "宣讲 PPT",
    "theme": "business",
    "stepId": "D_SLIDES",
    "data": {
      "artifactType": "PRESENTATION",
      "title": "宣讲 PPT",
      "theme": "business",
      "slides": []
    }
  }
}
```

渲染规则：

- `preview.available=false`：展示“预览生成中/尚未生成”，不要展示空白编辑器。
- `preview.available=true` 且 `preview.type=slides`：渲染 PPT 预览编辑器。
- `preview.available=true` 且 `preview.type=doc`：渲染文档预览编辑器。

### 精修动作

```json
{
  "adjustments": {
    "available": true,
    "stepId": "D_SLIDES",
    "phase": "confirm2",
    "actions": [
      {
        "key": "theme",
        "label": "切换主题",
        "type": "select",
        "target": "preview.data.theme",
        "options": ["business", "tech", "campaign", "minimal"],
        "currentValue": "business",
        "endpoint": "/api/v1/tasks/task-1/workspace/steps/D_SLIDES/preview",
        "method": "PUT"
      }
    ]
  }
}
```

保存结构化预览：

```http
PUT /api/v1/tasks/{taskId}/workspace/steps/{stepId}/preview
Content-Type: application/json

{
  "previewData": {
    "artifactType": "PRESENTATION",
    "title": "新标题",
    "theme": "tech",
    "slides": []
  }
}
```

自然语言精修：

```http
POST /api/v1/tasks/{taskId}/workspace/steps/{stepId}/preview/refine
Content-Type: application/json

{
  "instruction": "把整体风格改得更适合销售团队，减少技术细节",
  "source": "workspace",
  "clientId": "desktop-web"
}
```

### 时间轴

```json
{
  "timeline": [
    {
      "timestamp": "2026-05-03T09:30:00Z",
      "type": "STEP_PREVIEW_READY",
      "title": "预览已生成",
      "message": "步骤预览已生成",
      "level": "warning",
      "stepId": "D_SLIDES",
      "source": "agent",
      "sourceDisplay": "Agent",
      "metadata": {
        "stepId": "D_SLIDES"
      }
    }
  ]
}
```

渲染建议：

- 时间轴只展示事件，不要用它反推主进度。
- 来源展示使用 `sourceDisplay`。
- 排序建议按 `timestamp` 升序或倒序固定一种，不要每次更新改变顺序。

## 前端状态机建议

页面可以分为 4 个 UI 状态：

| 状态 | 触发条件 | 展示 |
|---|---|---|
| `loading_snapshot` | 已拿到 `taskId`，快照接口未返回 | 骨架屏/加载态 |
| `ready` | 快照接口成功返回 | 渲染完整工作台 |
| `syncing` | 已渲染快照，SSE 连接中 | 页面保持可用，可显示轻量同步状态 |
| `error` | 快照接口失败 | 展示错误和重试按钮 |

不要在 `loading_snapshot` 状态展示业务空态。

## 多端同步联调用例

### 用例 1：卡片进入工作台首屏不空白

1. 从飞书卡片点击“前往工作台”。
2. URL 带 `taskId`。
3. 前端立即请求 `GET /api/v1/tasks/{taskId}/workspace`。
4. 快照返回后首屏展示标题、进度、步骤、确认区。

验收：

- 不出现“暂无任务内容”等业务空态闪烁。
- 进度条使用 `progress.percent`。

### 用例 2：confirm1 到 confirm2 进度不回退

1. 任务进入 confirm1。
2. 快照中 `progress.phase=confirm1`，`percent=40`。
3. 用户点击确认。
4. 预览生成中快照为 `preview_generating`，`percent=50`。
5. confirm2 卡片/工作台出现后，快照为 `confirm2`，`percent=60`。

验收：

- 不出现 60% 回到 40%。
- 步骤文案显示“预览待确认”，而不是泛泛的“等待确认”。

### 用例 3：移动端确认，桌面端同步

1. 桌面端打开工作台并建立 SSE。
2. 移动端点击确认或精修。
3. 桌面端收到 SSE `workspace` 事件。
4. 桌面端替换完整 `TaskWorkspaceView`。

验收：

- 双端标题、进度、确认区、预览区一致。
- 时间轴新增事件来源能显示“工作台/飞书卡片/Agent”。

## 常见问题

### 为什么 SSE 也会先发 `snapshot`，还要请求快照接口？

首屏需要稳定、可控的加载链路。HTTP 快照更适合首屏渲染和错误重试；SSE 适合页面已可用后的实时更新。两者都返回完整视图，前端实现可以保持简单。

### 为什么不能根据步骤状态自己算进度？

confirm1 和 confirm2 都是同一个步骤的 `WAIT_CONFIRM`，仅靠 `status` 无法区分阶段。后端已经提供 `progress.phase`、`progress.percent` 和 `steps[].phase`，前端应直接使用这些字段。

### 快照返回前应该展示什么？

展示骨架屏、加载条或“正在同步任务状态”。不要展示“暂无内容”“暂无预览”“没有任务”等业务空态。
